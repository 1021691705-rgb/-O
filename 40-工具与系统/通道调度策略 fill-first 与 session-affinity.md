# 通道调度策略 fill-first 与 session-affinity

## 一句话总结

`fill-first` 和 `session-affinity` 是用于账号池、通道池或上游实例池的调度策略：前者减少频繁轮换，后者让同一个会话尽量粘到同一个上游，TTL 控制粘性保留时间。

## 来源

- 用户补充：2026-07-13
- 原始配置：
  - `fill-first`
  - `session-affinity: true`
  - `session-affinity-ttl: 168h`

## 参数解释

| 参数 | 含义 | 作用 |
| --- | --- | --- |
| `fill-first` | 优先把请求打到同一个可用账号/通道，不频繁轮换 | 提高会话连续性，减少账号池抖动 |
| `session-affinity: true` | 同一个会话尽量粘到同一个上游账号/实例 | 避免上下文、限流状态或登录态来回切换 |
| `session-affinity-ttl: 168h` | 会话粘性保留 168 小时，也就是 7 天 | 控制会话到上游实例的映射缓存时长 |

## 适合场景

- 多账号池、代理池、模型 API 上游池、网关路由、AI 工具通道调度。
- 需要维持同一会话上下文或稳定通道。
- 希望减少频繁切号带来的风控、限流或状态不一致。

## 风险与注意

- `fill-first` 可能导致单个账号或通道压力过高。
- `session-affinity-ttl` 太长时，异常账号可能持续影响同一会话。
- 需要配合健康检查、熔断、失败重试和容量上限。
- 7 天适合稳定会话；如果账号状态经常变化，可以缩短 TTL。

## 建议配置思路

- 默认启用 `session-affinity`，让同一会话保持稳定。
- `fill-first` 要和健康检查、失败转移一起使用。
- TTL 按业务周期设置：长期项目可用 `168h`，短任务可更短。
- 记录会话到上游的映射时，不保存敏感 token、cookie 或账号密钥。
