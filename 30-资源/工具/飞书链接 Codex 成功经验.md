# 飞书链接 Codex 成功经验

## 一句话总结

通过飞书企业自建应用的长连接模式，把飞书消息转发给本地 Codex，再把 Codex 的回复发回飞书，实现了在飞书里直接调用本地 Codex 的工作流。

## 成功的关键

- 使用飞书长连接模式，而不是依赖临时公网回调地址。
- 本地启动 Node.js 服务，负责接收飞书事件、调用 Codex、发送回复。
- 飞书开放平台里订阅消息事件，并授予收消息和发消息权限。
- 权限或事件配置变更后，重新创建并发布应用版本。
- 敏感信息只放在 `.env`，不写入笔记、代码仓库或聊天记录。

## 推荐架构

```text
飞书消息事件
-> 本地长连接服务
-> 消息解析与命令路由
-> Codex 处理普通文本任务
-> 飞书消息接口返回结果
```

如果以后扩展生图能力，可以在命令路由里增加 `/生图`，再调用图像生成接口，并把结果上传回飞书。

## 飞书端配置要点

- 创建企业自建应用。
- 开启机器人能力。
- 把机器人添加到目标用户或群聊。
- 事件订阅方式选择“长连接”。
- 订阅消息接收事件：`im.message.receive_v1`。
- 授权接收消息权限：
  - `im:message.p2p_msg:readonly`
  - `im:message.group_at_msg:readonly`
  - `im:message:readonly`
- 授权机器人发送消息权限：
  - `im:message:send_as_bot`
- 每次改权限、事件或订阅模式后，都要发布新版应用。

## 本地服务配置要点

`.env` 中至少需要这些配置，但不要在笔记里保存真实值：

```env
FEISHU_APP_ID=
FEISHU_APP_SECRET=
FEISHU_BOT_NAME=Codex
CODEX_COMMAND=codex.cmd
CODEX_WORKSPACE=
CODEX_TIMEOUT_MS=600000
```

## 消息处理经验

- 只处理文本消息，忽略图片、文件等非文本消息。
- 群聊里最好要求明确提及机器人，避免误触发。
- 用会话 ID 保存 Codex 线程，保持上下文连续。
- 同一个会话里的任务排队执行，避免多个 Codex 任务同时改同一份工作区。
- 回复飞书前可以先发送“已收到，正在处理”，降低等待时的不确定感。

## 常见问题排查

### 没有回复

- 检查本地 Node.js 服务是否正在运行。
- 检查运行日志里是否显示长连接成功。
- 检查飞书应用是否已经发布最新版。
- 检查机器人是否被添加到当前聊天。

### 能连接但收不到消息

- 确认事件订阅方式是长连接。
- 确认订阅了 `im.message.receive_v1`。
- 确认权限变更后重新发布过应用。
- 确认没有另一个本地服务实例正在消费同一个应用的事件。

### 收到消息但发不出回复

- 检查 `im:message:send_as_bot` 权限。
- 检查发送消息接口的错误日志。
- 检查接收 ID 类型是否匹配聊天类型。

## 这次经验带来的启发

- 对本地工具来说，长连接比临时公网回调地址更稳定。
- 先打通最小链路，再加复杂能力，成功率更高。
- 密钥、运行日志、临时票据都要和知识笔记分开保存。
- 能自动化的一键启动脚本很重要，否则成功经验很难复现。
