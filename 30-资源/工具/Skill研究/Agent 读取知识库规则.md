# Agent 读取知识库规则

## 一句话总结

Agent 执行任务前，先按固定路径检索知识库，读取最相关的 1-5 篇笔记，再结合当前任务上下文执行；知识库规模变大后，再升级为 RAG。

## 核心概念

| 概念 | 作用 | 当前做法 |
|---|---|---|
| 知识库 | 长期保存流程、Skill、工具经验、复盘和规则 | Obsidian Markdown |
| 检索系统 | 帮 Agent 找到该读哪几篇笔记 | 入口页 + 工作流地图 + 关键词搜索 |
| 上下文 | 当前这次任务的输入、图片、文件、限制和聊天要求 | 每次任务临时提供 |
| 记忆 | 用户长期偏好、常用规则、协作习惯 | 写进说明书、规则页和 Skill 笔记 |
| RAG | 检索相关知识后再生成或执行 | 知识库变大后再做 |

## 当前阶段策略

现在知识库规模还小，先不用复杂向量数据库或 RAG。

当前采用：

```text
用户任务
-> 判断任务类型
-> 读取首页 / 使用说明书 / 工作流地图
-> 按关键词和阶段找到相关 Skill
-> 读取 1-5 篇相关笔记
-> 比较触发场景和输出
-> 执行任务
-> 必要时把新经验写回知识库
```

### 3. 相似 Skill 的处理

如果多个 Skill 都像能用：

1. 先比较“触发场景”。
2. 再比较“输出”。
3. 再看用户明确要求。
4. 仍不确定时，先问用户，不直接执行。

示例：

| 容易混淆 | 判断方式 |
|---|---|
| 电商详情页方案 vs AI生图 Prompt | 前者规划页面结构，后者把已确认画面转成生图提示词 |
| Photoshop 裁切导出 vs 图片质检 | 前者处理图片，后者检查图片问题 |
| 设计项目开项 vs 岗位任务分配 | 前者明确项目边界，后者分配执行任务 |

## 读取数量限制

默认只读取最相关的 1-5 篇笔记。

建议：

- 小任务：读 1-2 篇。
- 中等任务：读 3-5 篇。
- 大项目：先读总览和工作流，再按阶段逐步读取。

不要为了“更全面”一次性读取太多笔记，否则容易稀释当前任务上下文。

## 上下文和知识库怎么区分

当前任务上下文优先级最高，包括：

- 用户刚发的图片、文件、链接、文案。
- 用户本次明确要求。
- 本次输出格式。
- 当前限制和截止条件。

知识库提供长期规则，包括：

- 流程步骤。
- 常用 Skill。
- 质量标准。
- 历史经验。
- 风险提醒。

如果当前用户要求和知识库规则冲突，先提醒用户确认。

## 记忆怎么保存

不要把“记忆”只放在聊天里，重要偏好要写进知识库。

适合写入记忆的内容：

- 常用工作路径。
- 常用 Skill。
- 命名规则。
- 同步规则。
- 常用网站和工具。
- 用户明确说“以后都这样”的偏好。

不适合写入记忆的内容：

- 密钥、token、cookie。
- 临时验证码。
- 未确认的账号信息。
- 客户隐私和敏感素材。

## 什么时候升级 RAG

现在先不做复杂 RAG。

当出现以下情况，再升级：

- 笔记超过 300 篇。
- 同类 Skill 超过 30 个。
- 搜索关键词经常找不到正确笔记。
- 同一问题需要跨 10 篇以上笔记整合。
- Agent 经常用错 Skill 或漏读关键上下文。

升级方向：

```text
Markdown 知识库
-> 元数据和关键词索引
-> 本地全文搜索
-> 向量检索 / embedding
-> RAG 召回相关笔记
-> Agent 结合当前任务执行
```

## 当前要先做好的事

- 每篇正式笔记有清晰标题。
- Skill 笔记写清触发场景、输入、输出和相关链接。
- 总览页保持短，只做导航。
- 新笔记必须回答关联性问题。
- 定期跑 [[30-资源/工具/Skill研究/知识库健康检查清单]]。
