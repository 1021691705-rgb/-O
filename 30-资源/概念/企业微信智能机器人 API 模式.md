# 企业微信智能机器人 API 模式

## 一句话总结

企业微信智能机器人 API 模式，是把企业微信消息接入外部 AI 系统的一种方式。它可以通过长连接或回调，把用户消息交给后端服务，再由后端服务调用 Agent、模型或业务系统生成回复。

## 适用场景

- 企业微信里接入 AI 助手。
- 企业微信群里使用多 Agent 协作。
- 不同群或用户绑定不同知识库、工作区或 Agent。
- 把企业微信作为内部工具入口，而不是让用户单独打开另一个系统。

## 两种接入方式

### 长连接

优点：

- 不需要公网回调域名。
- 不需要频繁维护回调 URL。
- 更适合本地或内网服务。

注意：

- 同一个应用不要多处同时消费事件。
- 服务需要保持在线。
- 要有健康检查和自动重连机制。

### URL 回调

优点：

- 传统 Webhook 思路清晰。
- 适合部署在公网服务器上的服务。

注意：

- 需要公网可访问地址。
- 可能需要配置 Token、Encoding-AESKey 等安全参数。
- 本地开发时容易遇到隧道不稳定或回调地址变化。

## 和 OpenClaw 的关系

OpenClaw 可以把企业微信作为一个消息通道。企业微信负责接收用户消息，OpenClaw 负责根据通道和绑定规则，把消息路由给不同 Agent。

推荐理解：

```text
企业微信消息
-> 企业微信智能机器人 API 模式
-> OpenClaw 企业微信通道
-> bindings 路由规则
-> 指定 Agent
-> 回复企业微信
```

## 和飞书 Codex 项目的关系

它和 [[飞书链接 Codex 成功经验]] 的思路相似：

```text
IM 工具
-> 消息事件
-> 本地或云端服务
-> Agent / Codex
-> 回复 IM 工具
```

区别在于：

- 飞书项目是自己写 Node.js 服务接 Codex。
- OpenClaw 方案是通过 OpenClaw 的 channel、agent、bindings 来完成路由和隔离。

## 复刻时重点检查

- [ ] 企业微信机器人是否创建为 API 模式。
- [ ] 接入方式是长连接还是 URL 回调。
- [ ] Bot ID 和 Secret 是否配置正确。
- [ ] OpenClaw 企业微信通道是否安装。
- [ ] Gateway 是否重启。
- [ ] Agent 是否创建。
- [ ] bindings 是否正确绑定到 Agent。
- [ ] 不同 Agent 的 workspace 和 memory 是否隔离。
