# 规则与规格

> Stable workflow principles for image generation, image editing, reverse prompting, prompt writing, reference handling, text rendering, and knowledge-base recording. Do not append small patch rules repeatedly; merge new lessons into the nearest existing principle.

## 本地执行口径

- 本页是视觉提示词知识库的稳定规则页，具体任务仍以用户要求和当前 Skill 为最高优先级。
- 有向量检索能力时使用分层向量检索；当前环境没有该能力时，使用 Obsidian 搜索或 `rg` 完成同等层次的关键词检索。
- 有 Wiki lint 工具时运行该工具；当前环境没有时，至少执行 Markdown 格式、内部链接和重复词块检查。
- 只采用与当前任务有关的词块，不因命中知识库而机械堆叠提示词。

## 1. Knowledge Base Structure

The visual prompt knowledge base uses three main pages:

- `Visual Phrase Blocks` → reusable prompt phrases for image generation, e-commerce poster layout, typography, information modules, scene language, and image-editing controls
- `Reverse Prompt Templates` → reference-image analysis templates and reverse-prompt checklists
- `Rules and Specifications` → model, size, quality, process, reference handling, and recording rules

Rules:

- Do not create new top-level modules by source, industry, quality label, platform, page type, or temporary origin.
- Treat carousel main images, Amazon A+ images, detail pages, Taobao detail pages, marketplace modules, and similar commercial page formats as e-commerce poster variants, not separate categories.
- Store the reusable visual mechanism instead: poster structure, information hierarchy, comparison module, specification band, product hero layout, icon callout, scene proof, or series consistency.

## 2. Recording Language and Content Boundaries

- Record reusable phrase blocks and templates in English by default.
- Translate into Chinese only when the user asks for a Chinese explanation.
- Do not store full finished prompts.
- Do not store long project reviews.
- Do not store exact image text, exact numeric values, brand names, prices, or product-specific one-off details.
- Store reusable visual structure, visual method, typography effect, layout logic, lighting method, material expression, or editing control instead.

## 3. Visual Task Workflow

Standard flow:

```text
Understand the request → Search the knowledge base → Read relevant phrase blocks → Write the prompt → Generate / edit / reverse → Review the result → Deposit transferable reusable phrases if useful
```

Mandatory principles:

- Run knowledge-base search before serious image generation, image editing, reverse prompting, or prompt optimization. Use vector search when available; otherwise use Obsidian search or local full-text search.
- Use layered retrieval; the retrieval step is incomplete unless it records queries and results for four layers: task-domain/subject-class, visual-mechanism, quality-control, and task-specific constraints.
- Concrete nouns from the user request should narrow retrieval, not replace mechanism retrieval. Product names, props, clothing, actions, locations, colors, platform names, or page types must not be the only query.
- If the task-specific layer misses, broaden upward and read stable sections directly before writing the prompt or deciding on deposits. Use broad section-level retrieval such as portrait, product, scene/space, composition, lighting, material, layout/typography, image quality, commercial quality, interaction, or edit-control.
- Search results are only the entry-finding step, not the usefulness test. Before writing a visual prompt or generating an image, read the stable Wiki pages/sections that match the broad mechanisms and extract adopted phrase blocks from the page text.
- For normal commercial image prompts, do not proceed from a low-search-hit state alone. First list or internally record a compact set of adopted phrase blocks from at least three relevant mechanism groups when available, such as subject interaction, product presentation, composition/layout, lighting/material, portrait/lifestyle quality, text/module clarity, or artifact prevention. If enough adopted blocks cannot be found after reading stable sections, report the exact missing mechanism group instead of implying the knowledge base was searched successfully.
- Agents should record queries for each layer, hit/miss results, adopted phrase blocks, stable sections read after misses, and deposit suggestions when content is uncertain or potentially polluting. Ordinary accepted blocks can be written after duplicate and quality checks.
- Use only relevant phrase blocks; do not overload the prompt with decorative words.
- The final prompt must clearly describe the image structure, not only mood or style.
- After completion, run the phrase-deposit closing check when reusable content appears. Ordinary accepted micro-mechanisms are written directly after duplicate and quality checks; only uncertain, too-specific, near-duplicate, or potentially polluting phrases need user review.

## 4. Prompt Writing Structure

Recommended prompt order:

```text
Main subject + scene + composition + camera angle + lighting + color mood + material/texture + style + image quality + constraints
```

For commercial images:

```text
Subject + commercial purpose + layout/composition + lighting + material + product/portrait treatment + background + concrete image-quality mechanisms + constraints
```

For reference-based generation:

```text
New subject + reference composition + reference layout structure + reference camera angle + reference lighting + reference mood + required changes + constraints
```

## 5. Model, Size, and Quality Rules

- Use the image model specified by the active tool or skill.
- Choose size based on final use; infer aspect ratio if the user does not provide one.
- Detail page sections usually need vertical or long-form composition.
- Poster, product-page, marketplace, and key visual images usually need portrait, square, or long-form composition according to final placement; do not turn platform names into knowledge-base categories.
- Product catalog images usually need square or clean centered framing.
- Do not invent model-specific flags unless the active tool supports them.
- Use high-quality mode for final commercial images and draft mode only for exploration.
- Prioritize clear subject, clean composition, realistic lighting, and commercial finish.
- Do not sacrifice product fidelity for decorative style.

## 6. Reference and Reverse-Prompt Rules

A reference image can provide:

- composition and layout structure
- camera angle and lens feeling
- lighting and shadow quality
- color palette and atmosphere
- visual rhythm and commercial finish
- typography style and text hierarchy
- icon, label, chart, panel, indicator, or information-module structure
- foreground, midground, background relationships
- concrete commercial image-quality mechanisms

A reference image should not automatically provide:

- exact text content unless requested
- exact numeric values
- brand names, logos, prices, or copyrighted characters
- unrelated products, irrelevant people, or incidental objects

Reverse-prompting must capture:

- exact count, position, arrangement, state, and meaning of important visible elements
- typography style, font weight, case, texture, edge treatment, spacing, and readability
- icon style, stroke weight, outer shape, internal symbol, and label relationship
- module structure, indicator states, percentage/range logic, comparison marks, and repeated elements
- concrete commercial image-quality mechanisms whenever relevant: how the subject is grounded, how contact shadows touch the surface, how reflections fade, how materials respond to light, how mist, glow, liquid, smoke, particles, or motion effects blend into the scene, how text zones stay readable, how card/module edges stay clean, and what artifact risks must be avoided

Core checks:

```text
If someone only reads the final prompt, can they imagine the same image structure as the reference?
Have all important counts, positions, states, meanings, typography effects, and icon details been described?
Have the quality direction been described as concrete visual behavior rather than generic words such as premium, high quality, or realistic alone?
```

## 7. Product Image Rules

When a product image is provided:

- Treat it as the product identity anchor.
- Preserve the real product appearance through the image input, not through long product-consistency wording in the prompt.
- Do not over-describe the product's physical details in the prompt.
- Let the product image carry exact shape, color, logo, label, buttons, texture, screen content, packaging text, and proportions.
- Use the prompt mainly to control scene, composition, camera angle, product position, product scale, pose/usage relationship, lighting, mood, layout, information modules, and concrete image-quality mechanisms.
- Avoid product-consistency protection wording such as `do not redesign the product`, `keep all original features`, `preserve exact product identity`, or `maintain recognizable product appearance` in final generation prompts.
- For multi-image main-image sets, write each image as a separate positive visual design with its own camera angle, product orientation, product scale, scene, model pose if any, and text zone. Shared style means shared palette, typography, lighting, and quality mechanisms, not identical product angle.
- Do not redesign the product unless requested.

## 8. Text Rendering Rules

- AI image models may not render text perfectly.
- Use short text only when necessary.
- For complex copy, foreign-language copy, or strict commercial typography, prefer a no-text image with reserved blank areas for later design.
- If text must be generated, specify exact text and require no extra characters.
- Always check spelling, duplication, and readability after generation.
- Do not deposit specific generated or reference-image text into the knowledge base; deposit reusable text layout and typography structure instead.

## 9. Image Editing Rules

- Editing prompts should state what changes and what stays unchanged.
- Use `change only...` when the edit scope must be strict.
- Preserve subject identity, geometry, framing, and lighting unless the user asks otherwise.
- For product edits, do not restyle or redesign the product.
- For text replacement, keep typography hierarchy and layout unless the user asks for redesign.

## 10. Iteration Rules

When a result is wrong, adjust only the key failure point:

- wrong subject → clarify the subject
- wrong composition → strengthen layout and position terms
- wrong camera angle → add camera/perspective terms
- weak lighting → add lighting direction and shadow terms
- weak commercial quality → add concrete quality mechanisms, such as grounding, contact shadows, material response, effect-light integration, text-zone clarity, module-edge clarity, or artifact constraints
- product distortion → reduce product description and strengthen preservation rules
- bad text → reduce generated text or reserve blank space

Do not rewrite the entire prompt blindly after every failed image.

## 11. Reusable Phrase Deposit Rules

After image generation, image editing, reverse prompting, or prompt optimization:

- Check whether transferable reusable phrase blocks appeared.
- Use transferability as the standard: if the expression can still help after changing the product, scene, or project, generalize it and save it.
- Big categories are retrieval entrances, not deposit results. Do not deposit standalone broad categories or vague taste labels.
- Transferable content should be micro-mechanisms: composition behavior, layout structure, lighting behavior, material/texture response, typography treatment, icon/information module behavior, product-role/subject-interaction language, edit-control language, reverse-check rules, actionable quality-control behavior, and artifact-prevention controls.
- A candidate phrase must be specific enough to control generation. Prefer phrases that include visual behavior, position/context, subject relationship, material or lighting response, information behavior, quality problem prevention, or commercial purpose.
- Reject standalone big words or vague labels such as broad subject classes, generic style names, `premium`, `dynamic`, `futuristic`, `high-end`, `realistic`, `clean`, `soft lighting`, or `commercial finish` unless the phrase includes the operational mechanism.
- Do not save exact image text, exact numeric values, exact product names, brand-specific details, or one-off incidental details.
- Write phrase blocks for vector search retrieval: prefer short mechanism phrases shaped as `use case + visual mechanism + position/context + purpose`; use a few long structure blocks only when the full visual relationship is important.
- Directly save accepted phrase blocks after duplicate and quality checks. Ask the user to review only uncertain, too-specific, near-duplicate, or potentially polluting phrases.
- Prefer merging into the three existing main pages.
- Do not create duplicate modules.
- Use vector search plus local exact/near-duplicate checking before writing.
- Run wiki lint after meaningful updates when available; otherwise run Markdown format, internal-link, and duplicate checks.
- Report the deposit result in the final reply using one exact standard format: `词块沉淀：已检查，新增 X 条。` or `词块沉淀：已检查，已有相似词，无需新增。`.

## 12. Rule Maintenance Rules

- Do not append a new standalone rule for every small correction.
- When a new lesson appears, first merge it into the nearest existing principle.
- Rewrite the relevant section if needed, instead of stacking patch-like bullets.
- Keep rules short, stable, and operational.
- If a rule becomes too specific, generalize it or move it into the relevant phrase page.
